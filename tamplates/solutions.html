🔧 bugfarm00x Solutions

How to identify and fix each vulnerability.

#1 — SQL Injection

SQLi

🕵️ How to find it

In main/views.py:UserLogin, the login query is built using an f-string:

query = f"SELECT id FROM auth_user WHERE username = '{username}' AND password = '{password}'"

User input is directly concatenated into the SQL statement with no sanitization or parameterization.

⚔️ Exploitation

Enter ' OR '1'='1 as the username and anything as the password. The query becomes:

SELECT id FROM auth_user WHERE username = '' OR '1'='1' AND password = 'x'

Since '1'='1' is always true, the query returns the first user (often admin), logging you in without knowing their password.

🛠️ The Fix

Use parameterized queries to separate SQL logic from data:

cursor.execute(
    "SELECT id FROM auth_user WHERE username = %s AND password = %s",
    [username, password]
)

Or better yet, use Django's ORM which handles this automatically:

user = authenticate(request, username=username, password=password)

#2 — Reflected XSS

XSS

🕵️ How to find it

In tamplates/home.html, the search query is rendered with the |safe filter:

<input ... value="{{ request.GET.q|safe }}">

Django normally auto-escapes HTML in template variables. The |safe filter disables escaping, allowing arbitrary HTML/JavaScript injection.

⚔️ Exploitation

Visit /feed/?q=<script>alert(document.cookie)</script>

🛠️ The Fix

Remove the |safe filter — Django's default auto-escaping handles XSS:

<input ... value="{{ request.GET.q }}">

If you must allow some HTML, use a sanitization library like bleach instead.

#3 — Insecure Direct Object Reference (IDOR)

IDOR

🕵️ How to find it

In main/views.py:UserProfile, there's a detail view that takes a pk from the URL:

class UserProfile(LoginRequiredMixin, DetailView):
    model = Profile
    ...

Compare with MyProfile which always returns the current user's profile via get_object() — UserProfile has no such check.

⚔️ Exploitation

Access /profile/1/ while logged in as a different user. You can view any user's profile by changing the ID.

🛠️ The Fix

Add an ownership check in the view:

class UserProfile(LoginRequiredMixin, DetailView):
    model = Profile
    template_name = 'myprofile.html'
    context_object_name = 'profile'

    def get_object(self):
        profile = super().get_object()
        # VULN FIXED: verify ownership
        if profile.user != self.request.user:
            raise Http404("Profile not found")
        return profile

#4 — Unrestricted File Upload

Unrestricted Upload

🕵️ How to find it

In main/models.py, the Poc.content field uses FileField with no validation:

content = models.FileField(upload_to='picpokcontent')

No file type, size, or content validation is performed during upload.

⚔️ Exploitation

Upload a .html file with embedded JavaScript, a .py file, or any executable. The file is stored under /media/picpokcontent/ and served directly.

🛠️ The Fix

Validate file type and content in the form or view:

import magic

def clean_content(self):
    file = self.cleaned_data['content']
    allowed_types = ['image/jpeg', 'image/png', 'image/gif', 'video/mp4']
    mime = magic.from_buffer(file.read(1024), mime=True)
    if mime not in allowed_types:
        raise forms.ValidationError('Invalid file type')
    if file.size > 10 * 1024 * 1024:  # 10MB
        raise forms.ValidationError('File too large')
    return file

#5 — Open Redirect

Open Redirect

🕵️ How to find it

In main/views.py:UserLogin, after successful login:

next_url = request.GET.get('next')
if next_url:
    return redirect(next_url)

No validation is performed on the redirect target — it accepts any URL, including external domains.

⚔️ Exploitation

Send a victim: /login/?next=https://evil.com/phish. After login, they're redirected to the attacker's site.

🛠️ The Fix

Validate the redirect URL using Django's built-in url_has_allowed_host_and_scheme:

from django.utils.http import url_has_allowed_host_and_scheme

next_url = request.GET.get('next')
if next_url and url_has_allowed_host_and_scheme(next_url, allowed_hosts={request.get_host()}):
    return redirect(next_url)
else:
    return redirect('createprofile')

#6 — Information Disclosure

Debug Mode

🕵️ How to find it

In picpok/settings.py:

DEBUG = True

When DEBUG=True, Django displays detailed error pages with stack traces, local variables, settings, and source code snippets.

⚔️ Exploitation

Visit a non-existent URL like /nonexistent or trigger an exception. The debug page reveals file paths, database configuration, and application internals.

🛠️ The Fix

Set DEBUG = False in production and configure proper error logging:

DEBUG = False
ALLOWED_HOSTS = ['your-domain.com']

# Configure logging instead
LOGGING = {
    'version': 1,
    'handlers': {
        'file': {
            'level': 'ERROR',
            'class': 'logging.FileHandler',
            'filename': 'django-errors.log',
        },
    },
    'loggers': {
        'django': {
            'handlers': ['file'],
            'level': 'ERROR',
        },
    },
}

#7 — Weak Password Storage

Weak Crypto

🕵️ How to find it

In main/forms.py:

def save(self, commit=True):
    user = super().save(commit=False)
    user.password = self.cleaned_data['password1']  # plaintext!
    if commit:
        user.save()
    return user

Instead of calling set_password() (which hashes), the form stores the password verbatim.

⚔️ Exploitation

Check the database directly (via SQLi or shell access):

SELECT username, password FROM auth_user;
 demo    | demo1234
 admin   | password123

🛠️ The Fix

Use Django's built-in set_password() which applies PBKDF2 hashing:

def save(self, commit=True):
    user = super().save(commit=False)
    user.set_password(self.cleaned_data['password1'])  # hashes with PBKDF2
    if commit:
        user.save()
    return user

Or simply use Django's UserCreationForm without overriding save() — it handles hashing automatically.
