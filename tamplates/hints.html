#1 — SQL Injection

Hint 1: Look at how the login query is built in main/views.py:UserLogin. What happens if you type a ' (single quote) in the username field?

Hint 2: Can you make the WHERE clause always evaluate to true? Try: ' OR '1'='1 as the username.

Hint 3: The query uses f-string interpolation. Any SQL you inject will be executed directly against the database.

#2 — Reflected XSS

Hint 1: Check the search bar on the feed page. The query parameter q is rendered with |safe.

Hint 2: Try entering: <script>alert('XSS')</script> in the search box.

Hint 3: The |safe filter in Django disables HTML escaping — user input is rendered as raw HTML.

Easy
#3 — IDOR

Hint 1: There are two profile views. One is safe, the other isn't. Compare MyProfile and UserProfile in main/views.py.

Hint 2: The UserProfile view accepts a pk parameter from the URL but never checks if it belongs to the current user.

Hint 3: Try accessing /profile/1/ while logged in as a different user. Can you see their profile?

Medium
#4 — Unrestricted File Upload

Hint 1: Look at the Poc model — the content field uses FileField with no validation.

Hint 2: Can you upload a .py file or an executable? What about an .html file with embedded scripts?

Hint 3: Uploaded files are stored under /media/picpokcontent/ and served directly without any type checking.

#5 — Open Redirect

Hint 1: After a successful login, the next parameter is checked. Where does it redirect to?

Hint 2: Try: /login/?next=https://evil.com — if you log in successfully, where do you end up?

Hint 3: No allowlist is used — any URL, including external domains, is accepted.

#6 — Information Disclosure

Hint 1: Check picpok/settings.py — what is DEBUG set to?

Hint 3: Combine this with the SQL injection vulnerability — can you extract plaintext passwords from the database?
